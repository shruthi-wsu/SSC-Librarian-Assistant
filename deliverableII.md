**1. Purpose of This Deliverable**

After finalizing your project scope and completing requirements modeling, the next step is to
translate those requirements into structured design representations and demonstrate
disciplined project planning.

Although this course emphasizes software quality, not full-scale documentation, we must
follow a principled software process. Design is a critical stage in that process. The quality of
your development process directly influences the quality of your final product.

The central objective of this deliverable is to produce a well-structured design that:

- Reflects your previously modeled requirements
- Establishes a foundation for implementation
- Enables evaluation of both product quality and process quality

This milestone also evaluates your engineering discipline, quality awareness, and early
implementation maturity.

**2. Scope of Design Work**

In a traditional software engineering course, design documentation may include:

- Data/Class Design
- Architectural Design
- User Interface Design
- Component-Level Design
- Deployment Design

Due to time constraints and the quality-focused nature of this course, you are required to
submit a focused subset of these artifacts.

All design models must clearly trace back to your requirements models (e.g., use-case
diagrams, user stories, functional requirements).

**3. PDF Submission Requirements**

Submit one combined PDF including the following sections:

**A. Architectural Design (High-Level)**

Provide a brief component diagram that:

- Identifies major system components
- Illustrates communication and relationships
- Reflects overall system structure


If you applied an architectural pattern (e.g., MVC, layered architecture, client–server,
microservices, event-driven), you must:

- Clearly name the pattern
- Briefly justify its suitability

The architecture must demonstrate:

- Separation of concerns
- Logical modularization
- Alignment with requirements

**B. Component-Level Design (Detailed)**

Provide elaborated component diagrams for key components.

Each diagram must include:

- Component name
- Attributes (if applicable)
- Operations / methods
- Clearly defined interfaces
- Dependencies

Your design must demonstrate:

- Clear interface boundaries
- High cohesion
- Controlled coupling
- Traceability to functional requirements

If design insights required requirement revisions, document:

- What changed
- Why it changed
- Updated artifacts (if applicable)
**4. Diagram Requirements**
- Diagrams must be created using a UML tool (Lucidchart, Draw.io, Visual Paradigm,
StarUML, etc.)
- Hand-drawn diagrams are not permitted
- Diagrams must be readable and professionally formatted
- Use consistent naming conventions


**5. Video Submission (Required)**

In addition to the PDF, each team must submit a structured video.

**Video Requirements**

- Length: 10–12 minutes
- Format: MP
- Upload to YouTube as Unlisted
- One video per team
- Submit link on Canvas

**Required Video Structure**

**1. Project Overview (2–3 minutes)**
    - Project idea and motivation
    - Target users
    - Problem being solved
    - Major functional requirements
    - Major non-functional requirements
**2. UML Diagrams & Design Decisions (Approx. 4 minutes)**

Present:

- Architectural component diagram
- Elaborated component diagrams

Explain:

- Component structuring decisions
- Architectural pattern used (if any)
- How design supports quality goals
- Interface design choices
- Key trade-offs considered

This section must demonstrate engineering reasoning.

**3. Feature Demonstration & Implementation Progress (Approx. 4 minutes)**

You must:

- Demonstrate developed features
- Show working components (live or walkthrough)
- Present system flow


- Show GitHub repository
- Show GitHub Projects Kanban board
- Highlight team contributions

You are required to use GitHub Projects and follow proper engineering practices:

- Issues creation
- Assignment to members
- Feature branches
- Pull requests
- Structured merging

Static diagrams alone are insufficient. Meaningful implementation progress must be shown.

**4. Quality Integration (1 minute)**

Briefly summarize:

- Key quality goals
- How design supports them
- How your workflow supports quality
- Next steps toward quality assurance
